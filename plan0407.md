## THE PROBLEM

"How can you create some sort of eval for RL envs such that when they run small runs they can easily hill climb on it to optimize the env quickly, then run big run." — OAI RL Lead

Translation: Before spending $1M on RL training, labs need a cheap way to measure environment quality, identify what's broken, fix it, and iterate until confident.

**Current state of the art:** Labs harvest tasks massively, filter brutally (95%+ rejection), validate in containers, profile difficulty, then iterate manually against reward hacking. "Labs spend more time debugging environments than training models in them" (SemiAnalysis). NeMo Gym's "sanity check" is "run your frontier model and see if scores make sense." There is no automated, multi-dimensional environment quality tool.

**The verification ceiling is THE bottleneck:** "Verification Limits Code LLM Training" (ICLR 2026) found synthetic unit tests are correct but incomplete in 53.6% of cases. OpenAI abandoned SWE-bench Verified after finding 59.4% of tasks have flawed tests. This is the problem.

## WHAT WE BUILD

envaudit combines the best ideas from 6 published papers into one tool, with a training-validated proof it works.

### Component 1: Verifier Scoring

**Source:** "Scoring Verifiers" (NVIDIA, Feb 2025, arXiv:2502.13820)

**What it does:** Quantitatively measures how well a task's test suite discriminates between correct and incorrect solutions. Not "are the tests weak?" but "how good is this verifier as a reward function?"

**Method:**
- For each task, generate N candidate patches of varying quality:
  - The gold patch (definitely correct)
  - LLM-generated genuine attempts (some correct, some not)
  - LLM-generated exploits (incorrect but designed to pass)
  - Mutated gold patches (subtly wrong — off-by-one, boundary errors)
  - Empty/trivial patches (definitely incorrect)
- Run each through Docker (same swebench harness we already have)
- Independently assess each patch's actual correctness (via gold patch similarity + LLM judge)
- Compute 4 metrics:
  - **Top-1 Accuracy**: Does the verifier rank the best patch highest?
  - **Spearman rho**: Does the verifier's ranking correlate with actual quality?
  - **Bottom-1 Accuracy**: Does the verifier reject the worst patch?
  - **MAE**: Mean absolute error between verifier score and true quality

**Output:** Per-task verifier quality score (0-1). Tasks where the verifier can't distinguish good from bad patches get low scores.

**Why this matters:** This is the NVIDIA-validated framework for evaluating verifiers. It's principled, quantitative, and directly measures what we care about: "will this test suite give correct RL reward signal?"

**Cost:** ~$0.10/task for LLM patch generation + Docker runs via GH Actions (free).

### Component 2: Adversarial Exploit Generation (Iterative)

**Sources:**
- EvolveCoder (March 2026, arXiv:2603.12698): Iteratively refines tests based on execution behaviors. Pass@1 drops from 43.8% to 31.2% as tests evolve.
- Code-A1 (March 2026, arXiv:2603.15611): Two models with opposing RL objectives — code LLM vs test LLM.
- TRACE (Jan 2026, arXiv:2601.20103): 54-category exploit taxonomy. Contrastive detection raises GPT-5.2 from 45% to 63%.

**What it does:** Goes beyond our current single Claude call. Iteratively generates exploits, checks which succeed, then generates HARDER exploits informed by what worked.

**Method:**
- Round 1: Generate 3 exploit patches per task using TRACE's 54-category taxonomy as structured guidance (what we already do, but with better prompting)
- Docker-verify each exploit
- Round 2: For tasks where no exploit succeeded, try harder strategies informed by execution logs from Round 1. For tasks where exploits DID succeed, analyze WHY they succeeded (which assertions didn't catch them)
- Round 3: Generate "EvolveCoder-style" adversarial patches — solutions that are ALMOST correct but subtly wrong in ways the test suite misses
- Each round is more targeted than the last

**Output:** Per-task exploit success rate (0-1) + specific exploit categories that work + specific verifier weaknesses identified.

**Why this is better than what we have:** Current system is one-shot. This is iterative and execution-informed. Each round uses information from previous rounds.

**Cost:** ~$0.30/task across 3 rounds + Docker runs via GH Actions.

### Component 3: Hybrid Verification Analysis

**Sources:**
- R2E-Gym (COLM 2025): Execution-based + LLM judge hybrid gives +8pp over either alone. Industry standard.
- SWE-RM (Dec 2025, arXiv:2512.21919): Open-source 30B reward model trained specifically for SWE-bench trajectory judging. Lifts Qwen3-Coder from 51.6% to 62.0%. Improves over execution-based feedback by 3pp.
- Meta Semi-Formal Agentic Reasoning (March 2026, arXiv:2603.01896): Structured reasoning achieves 93% accuracy on patch equivalence without execution.

**What it does:** Catches BOTH types of verifier failure:
- False positives (hackable): tests pass on incorrect solutions → reward hacking
- False negatives (overly strict): tests fail on correct solutions → wasted training signal (what OpenAI found in 59.4% of cases)

**Method:**
- For each candidate patch generated in Component 1+2:
  - Run execution-based verification (test suite pass/fail)
  - Run independent correctness judge (NOT raw Claude-as-judge). Three options, ranked by quality:
    1. **SWE-RM** (best): Open-source 30B reward model purpose-built for SWE-bench. Supports 256k context. Runs on Colab or via HuggingFace Inference API.
    2. **Semi-formal reasoning** (strong): Meta's structured prompting technique applied to Claude. Requires agent to construct explicit premises, trace execution paths, derive formal conclusions. 93% accuracy.
    3. **Claude-as-judge with self-consistency** (fallback): 3x sampling at different temperatures, majority vote. Uses multiver's self-consistency mechanism.
  - Compare execution verdict vs judge verdict
- Build confusion matrix per task:
  - Tests pass + judge says correct = TRUE POSITIVE (good)
  - Tests pass + judge says incorrect = FALSE POSITIVE (hackable)
  - Tests fail + judge says correct = FALSE NEGATIVE (overly strict)
  - Tests fail + judge says incorrect = TRUE NEGATIVE (good)
- Compute hybrid verifier quality: precision, recall, F1

**Output:** Per-task false positive rate (hackability) + false negative rate (strictness) + overall verifier F1.

**Why this matters:** This is the ONLY component that catches overly-strict tests. OpenAI found 59.4% false negatives. By using SWE-RM (purpose-built, not a generic LLM judge) or semi-formal reasoning (93% accuracy), we avoid the "Claude judging Claude" circularity problem.

**Cost:** ~$0.05/task with semi-formal reasoning. ~$0 with SWE-RM on Colab.

### Component 4: Difficulty Profiling

**Sources:**
- Online Difficulty Filtering (arXiv:2504.03380, Jan 2026): Tasks with p ≈ 0.5 maximize gradient signal. p ≈ 0 or p ≈ 1 → zero learning.
- DRIVE (arXiv:2511.06307, 2025): Training on 9K mixed-difficulty tasks caused 48.8% performance collapse vs. difficulty-focused approach.
- SWE-bench/experiments (GitHub): Per-instance results for 30+ model submissions.

**What it does:** Estimates how difficult each task is for the target model class and flags tasks with suboptimal difficulty.

**Method:**
- Download per-instance results from github.com/SWE-bench/experiments (30+ models, free)
- For each task: p = fraction of models that solved it
- Compute gradient signal: p(1-p)
- Flag: p < 0.05 (too hard, zero gradient) or p > 0.95 (too easy, zero gradient)
- Optional: for custom model, run a small sample (10-20 tasks) to estimate model-specific p

**Output:** Per-task difficulty score + gradient signal estimate + flag if outside optimal range.

**Why this matters:** DRIVE showed 48.8% collapse from wrong difficulty. DAPO filters p=0 and p=1 tasks (standard practice). This automates what labs do manually.

**Cost:** Free (existing data). Or ~$0.50 total if running a small model sample.

### Component 5: Iterative Environment Optimization Loop

**Sources:**
- EvolveCoder: Iterative test refinement improves discrimination
- HardTestGen (ICLR 2026): LLM-synthesized harder tests improve RL training. +11.22pp precision.
- ACECoder (ACL 2025): Fully automated test synthesis pipeline. +25% on HumanEval-plus.

**What it does:** For tasks flagged as weak, automatically generates improved tests, re-scores, repeats.

**Method:**
```
iteration = 0
while EQS < threshold and iteration < max_iterations:
    1. Score all tasks (Components 1-4)
    2. For each task with low verifier score:
       a. Identify specific weakness (from Component 2+3):
          - "Tests don't check edge case X"
          - "Tests accept patches that only fix formatting"
          - "Tests reject correct solutions that use approach Y"
       b. Generate targeted test augmentation (Claude, guided by specific weakness)
       c. Validate: augmented tests still pass with gold patch
       d. Validate: augmented tests block known exploits
    3. Re-score with augmented tests
    4. Report improvement: EQS_new vs EQS_old
    iteration += 1
```

**Output:** Augmented test suite + convergence curve (EQS per iteration).

**Why this IS the "hill climbing":** Each iteration measurably improves the environment. The lab sees the score go up. When it plateaus, they're ready to train.

**Cost:** ~$0.20/task/iteration for LLM test generation + Docker verification.

### Component 6: Training Validation (Colab Experiment)

**Sources:**
- HARDTESTGEN (ICLR 2026): RL on harder tests improves code generation
- GradAlign (Feb 2026): 50% corrupted rewards → 40pp catastrophic collapse
- DRIVE (2025): Strategic selection >> dataset size
- Unsloth GRPO Qwen3-4B Colab notebook (ready-made)

**What it does:** Proves envaudit works by actually training with and without filtering.

**Method:**
- Use a code RL dataset suitable for Colab (competitive programming or simplified SWE tasks with test-based rewards)
- Create two training sets:
  - ALL: full dataset
  - FILTERED: dataset minus envaudit-flagged tasks (low verifier score, hackable, wrong difficulty)
- Train Qwen3-4B with GRPO via Unsloth on Colab T4:
  - Group A: train on ALL for N steps
  - Group B: train on FILTERED for N steps (same compute budget)
- Evaluate both on held-out tasks
- Report: eval score difference, reward hacking rate difference, training stability

**Output:** "Training on envaudit-filtered tasks scored X% higher on held-out evaluation."

**Cost:** Free (Colab T4) + ~$0 (training uses local compute, not API).

## COMPOSITE ENVIRONMENT QUALITY SCORE

```
EQS(task) = w1 * VerifierScore       (Component 1: NVIDIA metrics)
          + w2 * (1 - ExploitRate)    (Component 2: adversarial testing)  
          + w3 * HybridF1             (Component 3: hybrid verification)
          + w4 * DifficultyScore      (Component 4: p(1-p) gradient signal)

where w1=0.35, w2=0.30, w3=0.20, w4=0.15 (tunable)

Per-task verdict:
  EQS > 0.7                          → KEEP
  EQS 0.4-0.7 AND fixable weaknesses → FIX (enter optimization loop)
  EQS < 0.4 OR unfixable             → DROP
```

## LESSONS FROM MULTIVER'S FAILURES (Critical for envaudit)

Multiver had specific problems we MUST avoid:

### 1. False Positive Rate was 41-51%
In balanced mode, 41.3% of SAFE code was flagged as vulnerable. In high-recall mode, 51.3%. This means if we apply the same architecture naively to envaudit, we'd flag 40-50% of GOOD tasks as bad — destroying useful training data.

**Mitigation for envaudit:** Docker verification is our ground truth. Unlike multiver (where FP meant a code review false alarm), our FP would be "envaudit says this task is hackable but no exploit actually passes Docker." We already have this — the 102/127 exploit patches that FAILED Docker verification are false positives from the LLM. Our real hackability rate is 25/127 (19.7% exploit success rate), which means ~80% of Claude's "high confidence" exploit predictions were wrong. This is fine because Docker catches them — but it means the LLM tier alone is unreliable and we MUST verify everything.

### 2. Pattern Threshold Calibration
Multiver's correctness agent flagged 98% of PyVul samples as FAIL because enterprise-grade thresholds (80% exception coverage) were inappropriate for real-world vulnerable code. The pattern tier dominated and was miscalibrated.

**Mitigation for envaudit:** Our pattern tier flags things like "only 1 assertion" or "high permissive ratio." These are softer signals than multiver's binary thresholds. But we must validate: do our pattern flags correlate with actual Docker-verified hackability? If pattern flags produce high FPR, lower their weight in the composite score.

### 3. RAG Hurt Recall
No-RAG got 90.3% TPR vs 78.3% with RAG (-12pp). RAG reduced false positives but also missed real vulnerabilities.

**Mitigation for envaudit:** Use RAG for exploit GENERATION (retrieve similar successful exploits to guide new attacks), not for JUDGING (where it might suppress valid findings). RAG should expand what we try, not constrain what we flag.

### 4. High Variance Across Runs
Balanced test TPR ranged 42-91% depending on the run. Self-consistency (3x sampling) helped but didn't eliminate variance.

**Mitigation for envaudit:** Docker verification eliminates LLM variance for the hackability dimension. The hybrid judge (Component 3) still has LLM variance — use SWE-RM (trained model, deterministic) or self-consistency (3x) for stability.

### 5. Ensemble Voting Must Be Tuned
Multiver's credibility weights (security 0.45, correctness 0.35, performance 0.15, style 0.05) were empirically validated on vulnerability detection. We can't just copy them.

**Mitigation for envaudit:** Start with equal weights, then tune against the Colab training validation. The weights that produce the best training outcome are the right weights.
